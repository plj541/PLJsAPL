Statements

ShellAPL is a compiler which creates .Net programs.  These programs use PLJsAPL to create APL objects and use the methods provide APL’s functional behavior.  APL expressions and assignment statements are supported.  This environment does not support multiple assignments or branch statements.  Statements with imbedded or multiple assignments would have to be rewritten.

 

Expressions display their results.
          3+⍳4
Assignment statements do not display.
          a←3+⍳4
Statements where an assignment is not the left most action, are not supported.
          7-a←3+⍳4
Branch to another line number, is also not supported.
          →L1

While GoTo is available in Visual Basic, it isn’t provided by the compiler.

 

ShellAPL supports statements which begin with colon ':'.  The statement must look like one of the following:
    :If ...

:ElseIf ...

:Else

:End If

   

:Do While ...     or     :Do Until ...

:Exit Do

:Loop

 

:Do

:Loop While ...       or     :Loop Until ...

 

:Select Case ...  or     :Select First ...

:Case ...

:Exit Select

:Case Else

:End Select

 

    :Return ...

The forms without the three dots, cannot have anything further on the line.  The forms followed by … require an APL expression. The If, While and Until words must be followed by a comparison which returns one value. Select Case is followed by a character value and the following Case statements are usually character constants.  Select First is followed by a value with at least one item, the subsequent Case statements can be either numbers or characters based on what you expect the value to contain.  Return is only useful when the last statement in the function isn’t the result of the function.

 

Finally, ShellAPL also supports statements written in .Net.  This release supports VB programs, so a line beginning with right bracket ']' is assumed to be Visual Basic.

 

Methods

 

Methods are created when you begin a line with ∇, followed by a name.

    ∇Fun

creates the method Fun.  You can edit a second time the same way, or use the

          )Edit Foo

system command to edit it once created.  You can also create methods with the system method ⎕Def.

          "⍵ ∇ ⍺+⍵" ⎕Def "Plus"

These are just definitions.  To use these methods you need to

    )Save

the workspace, enter

    )Compile

Then click on Build and Build WorkSpaces.

 

These methods take three forms:

 

          1)      One liners are essentially direct definition, just type  ∇Plus  and then

 

       +⍵ ∇ ⍺+⍵

 

    2)      Methods which begin with ∇ are much like classic APL, but only locals are itemized.

The rest of the function header is implied.  Type  ∇Minus  and then

 

       ∇;local;names

       ⍝ Comment on the monadic definition

       -⍵

       ∇

       ⍝ Comment on the dyadic definition

       ⍺-⍵

 

3)      Functions in the host language have multiple lines and do not begin with ∇.  However, to create one, you still type ∇ followed by a method name.  All methods, including host language functions can take advantage of the automatic renaming of reserved APL characters.

 

    ∆   will become the name of the function so it can be used for self reference.

    ⍵   Is the right argument.

    ⍺   If present, implies the dyadic definition.

    ⍶   If present, implies an operator.

    ⍹   If present, implies the operator is dyadic.

    ⍙   If present, implies the method is a property.  Properties behave like variables.

If an ⍺ is present, the definition is understood to be assignment.

    ⌶   If present, indicates that the operater can have an axis.  Axis is available in the

underlying DLL, but isn't yet recognized in the code generator, so can only be used from

       .Net methods.

 

The APL versions of methods recognize the last statement of the monadic and dyadic definitions is the result.  Any previous lines which are not assignments, : or ] will be displayed.  If you need to return from other than the last line, use the control structure.

:Return aValue

   

Every method can be one of four things; a function, a property, a monadic operator, or a dyadic operator.  These are described in the table below.

 

Function

Both the monadic:

          ⍴ ⍵

and dyadic definition:

          ⍺ ⍴ ⍵

Property

Is syntactically like a value. 

To return a value:

          ⍙

and to assign a value:

          ⍙ ← ⍺

OpMonadic

Creates an operator which takes one function as an argument.

Both the monadic:

          ⍶ ⍵

and dyadic definition:

          ⍺ ⍶ ⍵

OpDyadic

Creates an operator which takes two functions as arguments.

Both the monadic:

          ⍶ ⍹ ⍵

and dyadic definition:

          ⍺ ⍶ ⍹ ⍵

 

 

What Is Different?

 

Locals cannot be seen in called functions.  That is, a value can only be local or global.  However, globals are just available to the instance in which they appear.  An APL workspace is just a Class, so a link to it is just an object reference.  The syntax is just as it is in other .Net languages.  A name followed by a dot refers to an object which is an instance of the named class.

 

    a←⎕LinkTo "Samples"

    a.Report Data

 

Once you have an instance handle, you can use functions from any function in any workspace from any workspace.  Instance handles are not required when you are in the workspace with the same name as the compiled class.

 

In PLJsAPL, the only system values provided are ⎕AV, ⎕TS, ⎕CT and ⎕RL.  The only one likely to present much of an issue is ⎕IO.

 

⌹ does not support more rows than columns.

 

This is not APL2.  While enclose '⊂',  disclose '⊃' and each '¨' are available, and work like APL2, scalar function do not automatically extend to nested arguments.  I am researching this behavior, and may provide ∵ as an operator to provide this functionality.

 

Display of multiple dimensional arrays within multi dimensional arrays just shows the rank.  This is being researched, but nothing is currently available.

 

Simple strands are supported, but those involving parenthesis are not.

 

    "cat" "dog"

works, as does

    10 a 40

 

But,

    10 (2 3⍴⍳4) 40

does not, and neither does:

    10 (20 30) 40

 

In each case, you would need to assign the expression in parenthesis to a value and then type it as above.

 

At this point, the functionality users expect are in PLJsAPL.dll and CommandLine.dll.  PLJsAPL provides direct support for the .Net properties of APL, where as CommandLine provides the compiler from APL to .Net.  As a result, ⍎ is not of much interest without learning what CommandLine provides.

 

If you want to use APL from VB, type an APL expression in the same order as APL, but with parenthesis:

MonadicFn(Value)

Value(DyadicFn, Value)

 

PLJsAPL operators follow the same pattern:

Function(MonadicOp)

Function(DyadicOp, Function)

 

Both provide a derived function and thus can substitute for

MonadicFn  or    DyadicFn

in the syntax above.  A complete list of PLJsAPL functions and operators is in the link above.
